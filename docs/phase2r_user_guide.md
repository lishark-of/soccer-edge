# Phase 2-R 用户说明：串联纪律、解释层、CLV 与 CSV 回测可信度

JC Edge 的 Phase 2-R 目标是让赛前观察更像严厉交易者，而不是为了给出组合而强行组合。

本阶段仍然只做：

- 赛前观察信号
- 纸面模拟
- 风险诊断
- 回测复盘
- 用户可读解释

不提供真实投注、下单、支付、代购或自动化购彩能力。

## 1. 串联命中率纪律引擎

2串1 和 3串1 不只看赔率，也不只看 EV。

系统会同时检查：

- 每一腿是否有正向价值
- 每一腿风险是否过高
- 每一腿可信度是否足够
- 组合命中概率是否达到最低门槛
- 是否存在同场互斥
- 是否存在高赔率冷门腿
- 情报缺失是否过多
- 相关性折扣后是否仍有吸引力

默认纪律：

- 进取档 2串1 最低组合命中概率：20%
- 进取档 3串1 最低组合命中概率：12%
- 进取档单腿可信度门槛：50%
- 高赔率冷门默认不能作为串联核心

如果没有合格组合，App 应显示：

```text
暂无优秀串联观察
```

这不是坏结果，而是系统没有强行给出弱组合。

## 2. 高赔率冷门怎么读

高赔率冷门可能出现高 EV，但它不是稳健信号。

App 会把赔率较高的冷门标记为：

```text
高赔率冷门观察
```

它可以单独观察，但默认不适合作为串联核心。

原因是：

- 命中概率低
- 波动大
- 容易放大组合风险
- 缺少伤停、首发、天气或新闻时可信度更低

## 3. DeepSeek Pro 解释层

DeepSeek Pro 只用于解释，不参与计算。

可以解释：

- 为什么出现这个观察信号
- 为什么当前不建议串联
- 为什么比分/总进球只能作倾向参考
- 哪些情报缺失降低了可信度
- CLV 和回测结果怎么读

不能做：

- 改写概率
- 改写 EV
- 替代模型
- 改变候选筛选
- 生成购买指令
- 生成自动组合

默认关闭。

本地配置建议：

```text
JC_EDGE_DEEPSEEK_ENABLED=false
JC_EDGE_DEEPSEEK_API_KEY=你的 token
JC_EDGE_DEEPSEEK_MODEL=deepseek-chat
JC_EDGE_DEEPSEEK_MAX_INPUT_TOKENS=6000
JC_EDGE_DEEPSEEK_MAX_OUTPUT_TOKENS=800
```

token 只保存在本地 `.env.local`，不要提交到 Git。

## 4. CLV / 收盘赔率跟踪

CLV 用于赛后复盘，不用于赛前承诺。

含义：

- 如果赛前记录的赔率高于收盘赔率，说明市场后续可能支持了这个方向。
- 如果赛前记录的赔率低于收盘赔率，说明后续市场没有支持这个方向。
- 如果没有收盘赔率，状态就是等待复盘。

App 会显示：

```text
等待收盘赔率
```

这表示当前只能记录赛前赔率，不能计算 CLV。

## 5. 用户 CSV 回测可信度

用户导入 CSV 后，系统会评估这份数据适不适合用来回测。

评分维度：

- 样本量
- 赔率字段覆盖
- 赛果字段覆盖
- 时间跨度
- 数据来源类型

fixture / mock 数据最多只能给中等可信度，因为它们是演示数据，不是真实长期历史市场。

真实用户 CSV 想提高可信度，建议：

- 至少几百场比赛
- 覆盖 6 个月以上
- 胜/平/负赔率完整率接近 90%+
- 主客队进球或比分字段完整
- 联赛、主队、客队、日期字段清晰

## 6. App 第一屏应该怎么看

优先顺序：

1. Top 单关观察
2. Top 2串1观察或暂不组合原因
3. Top 总进球观察
4. Top 比分倾向
5. 严厉交易者结论

数据源、缺失情报、可信度细节属于诊断信息，不应盖过 Top 观察。

## 7. 正确心智

JC Edge 的目标不是告诉你“必须做哪一场”，而是告诉你：

- 哪些可以观察
- 哪些只是弱观察
- 哪些不应该串联
- 为什么不应该串联
- 缺什么情报
- 数据质量是否足够回测

如果系统说“不建议串联”，这通常是好事，说明它没有为了迎合组合而放松纪律。
